# 后续待办与规划

> 本文件是「检查未完成项 → 做出规划」的产出,记录当前已交付内容、评审循环结论,以及**有意暂缓**的后续项(按优先级)。
> 更新时间:2026-06-21。

## 一、当前状态

- 核心游戏(M0–M8)已完成:规则引擎、对局状态机、实时(Ably)、定时(QStash)、结算入库、战绩/排行榜、卡通前端,约 100+ 单测通过。
- 本轮新增并通过「安全 / 并发 / UX」三方评审循环(两轮,第二轮 must-fix 全为空):
  - **好友系统**:好友请求/接受/拒绝/列表/删除 + 邀请好友入房 + Ably `user:{id}` 实时通知(`@bullfighting/friends` 包、`/api/friends/*`、好友面板、迁移 0002/0003)。
  - **人机 PvE(练习模式)**:机器人决策 + 服务端驱动(不依赖 QStash)+ 临时练习房(仅 Redis、不入库、不动真实筹码)+ `/pve/[id]` 页面与大厅入口。

## 二、部署注意

- **数据库迁移**:好友功能依赖 `friendships` 表 —— 上线前需对 Neon 执行迁移 `0002_true_natasha_romanoff.sql` 与 `0003_solid_vindicator.sql`(`pnpm --filter @bullfighting/db db:migrate`)。
- **PvE**:无需新增环境变量。
- **QSTASH_TOKEN(既有)**:目前提供的是 `read_only` 令牌,会导致 PvP 的超时自动推进失效;PvE 不受影响(机器人为服务端同步驱动)。需替换为主令牌。

## 三、暂缓的后续项(按优先级)

### 高

- **接口限流**:登录/注册、好友请求、`/api/pve/start`、下注等敏感端点缺少按用户/IP 的限流(全站既有缺口,非本次功能引入)。建议用现有 Upstash Redis 做滑动窗口/令牌桶统一落地。
- **Ably token 能力收窄**已完成;遗留小项:PvP `roomId` 在写入 capability 前未与真实房间号规范化(`\d+` 容许前导零),纯防御性,非安全问题。

### 中

- **撮合队列**:排队用户掉线无清理 / 无 TTL,可能累积僵尸条目。
- **启动期环境变量校验**:密钥缺失目前到首次调用才报错,建议启动期 fail-fast。
- **入座筹码预校验**:`/api/rooms/[id]/join` 未在 API 层乐观拒绝筹码不足,失败需一次往返。
- **PvE 体验增强**:机器人逐帧动作节流(目前所有帧无延迟广播,机器人"瞬间"出手;已加"机器人行动中…/轮到你啦"提示作为过渡)。可加服务端帧间小延迟或客户端帧队列。
- **好友在线状态**:好友列表暂无在线/离线呈现(邀请为尽力而为,离线好友收不到)。可基于 Ably presence 在 `user:{id}` 频道实现。

### 低

- **客户端状态单调守卫**:PvE 下 HTTP 响应帧与实时中间帧可能短暂乱序闪烁(自我纠正,纯视觉)。可按 `(roundNo, phase)` 丢弃过期帧。
- **结算错误日志上下文**:用户中途被删导致结算回滚时,日志上下文偏薄。
- **reducer 出手者归属复验**:`findPlayer` 依赖更上游鉴权,可加防御性复验。
- **CSP 收紧**:目前为兼容 Next 运行时保留 `'unsafe-inline'`,UI 稳定后改 nonce。
- **e2e**:登录/注册渲染用例当前 `skip`,补齐后启用。
- **D1 TTL 刷新次序**:PvE 记录键 TTL 在状态保存前刷新,极端失败下记录略长于状态(无害)。
