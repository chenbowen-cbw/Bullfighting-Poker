# 安全设计与加固清单

> 适用:抢庄斗牛(虚拟娱乐筹码)全栈 TypeScript / Vercel 部署。
> 本文给出威胁面、已落地对策与待办建议,作为安全评审与上线检查表。

---

## 〇、安全总原则

- **服务端权威(Server-Authoritative)**:发牌、评牌、比牌、结算只在服务端完成,
  客户端只负责展示与输入。规则全部在纯函数引擎 `@bullfighting/core`,无网络/存储副作用,
  从架构上杜绝"改牌作弊"。
- **最小暴露**:私有信息(各家手牌)只对其本人可见;广播频道只承载公共状态。
- **纵深防御**:鉴权 + 输入校验 + 服务端权威 + 限流 + 安全响应头,任一层不作为唯一防线。
- **免合规风险**:筹码为虚拟娱乐筹码,不可兑换现金,不涉及支付 / KYC / 博彩合规。

## 一、威胁面与对策一览

| #   | 威胁                       | 风险                 | 对策(本仓库)                                             | 状态                          |
| --- | -------------------------- | -------------------- | -------------------------------------------------------- | ----------------------------- |
| 1   | 改牌 / 伪造结算            | 作弊、破坏公平       | 引擎纯函数 + 服务端权威;客户端不参与判牌/结算            | 已落地(架构)                  |
| 2   | 偷看他人手牌               | 信息泄露、作弊       | 手牌走**私有单播频道**,广播只发公共状态(见 §三)          | 设计约束(实时里程碑落地)      |

| 4   | 令牌伪造 / 篡改            | 冒充用户             | JWT(HS256)+ 强随机 `JWT_SECRET` + TTL;敏感操作再校验归属 | 已落地                        |
| 5   | 恶意 / 畸形输入            | 注入、崩溃、逻辑绕过 | 入口用 zod schema 严格校验,统一 `errorResponse` 收敛     | 已落地                        |

| 12  | 越权操作他人房间/座位      | 破坏对局             | 操作前校验"调用者属于该房间/座位/当前行动者"             | 已落地(房间域)/对局里程碑延伸 |

## 二、鉴权与会话(JWT)

- **签发**:登录成功后签发 JWT(HS256),载荷仅含必要字段(用户 id / 用户名),
  设置过期时间 `JWT_TTL`(默认 `2h`)。
- **校验**:受保护接口从 `Authorization: Bearer <token>` 取令牌,经
  `@bullfighting/auth` 验签 + 校验有效期,失败统一返回 401。
- **密钥**:`JWT_SECRET` 必须为长随机串,**生产从平台密钥注入**,不写入仓库。
  轮换密钥时可双密钥并行一段时间(旧验签 + 新签发)以平滑过渡。
- **归属再校验**:仅"已登录"不等于"有权操作该资源"。涉及具体房间/座位/对局的
  写操作,服务端需再次校验调用者确实是该资源的参与者 / 当前行动者(防越权)。
- **加固建议**:令牌尽量短时;前端存储优先 `httpOnly` Cookie 或内存,避免长期落
  `localStorage`;如引入刷新令牌,需配合撤销列表。

## 三、私有手牌:单播而非广播

实时通道(Ably)是本游戏信息安全的关键面。约束:

- **公共频道**只承载所有人可见的状态:座位、抢庄倍数、下注、亮牌后的结果、回合阶段。
- **私有手牌**通过**按用户的私有频道单播**下发,只有本人能订阅自己的手牌频道。
- **令牌权限收敛**:`/api/realtime/token` 颁发 Ably token 时,
  **服务端按当前用户限定可订阅/发布的频道能力(capability)**,
  禁止客户端自行订阅他人手牌频道。
- 亮牌前不在任何公共消息里出现他人底牌;亮牌阶段由服务端统一揭示。

> 这是"设计约束",在实时对局里程碑落地接口时必须遵守:**手牌永远不进广播**。

## 四、编排回调签名(QStash)

- 定时/延时推进对局通过 Upstash QStash 调用内部回调
  `${APP_URL}/api/internal/qstash/advance`。
- 该回调**必须校验 QStash 签名**(`Upstash-Signature`),使用
  `QSTASH_CURRENT_SIGNING_KEY` / `QSTASH_NEXT_SIGNING_KEY` 验签,
  拒绝任何未签名 / 验签失败的请求,**防止外部伪造推进**。
- 回调应**幂等**:用回合/状态版本号防重放(同一推进只生效一次)。
- 内部回调路径不对外暴露业务语义,且只接受来自 QStash 的签名请求。

## 五、安全响应头(middleware)

实现于 `apps/web/src/middleware.ts`,对所有匹配路由注入(不改 `next.config.mjs`,避免与前端配置冲突):

| 头                           | 值                                             | 作用                           |
| ---------------------------- | ---------------------------------------------- | ------------------------------ |
| `X-Content-Type-Options`     | `nosniff`                                      | 禁止 MIME 嗅探                 |
| `X-Frame-Options`            | `DENY`                                         | 防点击劫持(不可被 iframe 嵌入) |
| `Referrer-Policy`            | `strict-origin-when-cross-origin`              | 限制 Referer 泄露              |
| `Permissions-Policy`         | 关闭摄像头/麦克风/定位/topics                  | 收敛浏览器特性                 |
| `Cross-Origin-Opener-Policy` | `same-origin`                                  | 跨源开窗隔离                   |
| `Content-Security-Policy`    | 见下                                           | 限制资源来源,缓解 XSS          |
| `Strict-Transport-Security`  | `max-age=63072000; includeSubDomains; preload` | 仅 HTTPS 下发,强制 HTTPS       |

**CSP 基线**(`default-src 'self'`,放行同源 + Ably 的 `https:`/`wss:` 连接):

```
default-src 'self';
script-src 'self' 'unsafe-inline';
style-src 'self' 'unsafe-inline';
img-src 'self' data: blob:;
font-src 'self' data:;
connect-src 'self' https: wss:;
object-src 'none';
base-uri 'self';
form-action 'self';
frame-ancestors 'none';
```

> 收紧建议:当前为兼容 Next 内联运行时保留了 `'unsafe-inline'`。
> 前端 UI 里程碑稳定后,建议改为**基于 nonce 的 CSP** 并移除 `'unsafe-inline'`,
> 同时把 `connect-src` 收窄到具体 Ably 域而非泛 `https:`。HSTS 仅在 HTTPS 下下发,
> 避免污染本地 http 调试。

## 六、限流建议(待落地)

虚拟筹码无资金风险,但仍需防"撞库 / 滥刷 / DoS":

- **登录 / 注册**:按 IP + 账号双维度限流(如登录 5 次/分、注册 3 次/分),
  失败计数,触发后短时封禁或加验证码。利用已接入的 Upstash Redis 做计数窗口
  (令牌桶 / 滑动窗口),或用 `@upstash/ratelimit`。
- **建房 / 匹配 / 对局动作**:按用户限流,防脚本刷房与刷动作。
- **平台兜底**:Vercel 边缘对异常流量有一定防护,但应用层限流不可省。
- **统一 429**:超限返回 `429 Too Many Requests`,可带 `Retry-After`。

## 七、输入校验与错误处理

- 所有外部入口(API body / query)使用 **zod schema** 校验,类型与边界齐全
  (用户名长度/字符集、口令强度、数值范围如下注倍数/底分)。
- 错误统一经 `errorResponse` 收敛:`ZodError → 400`、领域错误按其 `status`、
  其余 `500` 且**不外泄堆栈/内部细节**;详细错误仅 `console.error` 落服务端日志。
- 永远不信任客户端传入的"结果性"字段(如牛型、输赢、金额),这些只由服务端计算。

---

## 八、数据与机密管理

- **口令**:scrypt 加盐哈希,绝不存明文;比对用恒定时间比较。
- **机密**:`DATABASE_URL`、`JWT_SECRET`、`UPSTASH_*`、`ABLY_API_KEY`、`QSTASH_*`
  等只走环境变量(Vercel 项目机密),不入库、不下发前端、不写日志。
- **仓库卫生**:`.env` / `.env.*`(除 `.env.example`)已 `.gitignore`;
  示例文件只放占位值。CI 不打印机密。
- **数据库**:Neon 连接强制 `sslmode=require`;最小权限账号;迁移用 `drizzle-kit`。

---

## 九、合规与公平性

- **免合规**:虚拟娱乐筹码,不可兑换现金,无支付/提现/KYC,不属于博彩。
- **公平性可验证**:洗牌用 Fisher–Yates;生产用加密级随机源 `cryptoRng`,
  测试用可复现 `mulberry32`(固定种子)。`packages/core/test/fairness.test.ts`
  以卡方/分布健全性检查验证洗牌均匀、无系统性偏置。
- **可审计**:支持把洗牌种子记入牌局以便事后审计(引擎已支持注入 RNG)。
