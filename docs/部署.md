# Vercel 部署指南(斗牛扑克)

> 形态:pnpm + Turborepo Monorepo,单 Next.js 15 应用(`apps/web`)部署到 Vercel。
> 依赖托管服务:Neon(Postgres)、Upstash(Redis + QStash)、Ably(实时)。

---

## 一、前置:开通三方服务

| 服务               | 用途                | 拿到的东西                                                              |
| ------------------ | ------------------- | ----------------------------------------------------------------------- |
| **Neon**           | Serverless Postgres | 连接串 `DATABASE_URL`(带 `?sslmode=require`)                            |
| **Upstash Redis**  | 匹配队列 / 限流计数 | `UPSTASH_REDIS_REST_URL`、`UPSTASH_REDIS_REST_TOKEN`                    |
| **Upstash QStash** | 定时/延时推进对局   | `QSTASH_TOKEN`、`QSTASH_CURRENT_SIGNING_KEY`、`QSTASH_NEXT_SIGNING_KEY` |
| **Ably**           | 托管 WebSocket 实时 | `ABLY_API_KEY`                                                          |

> 建议:为 **Preview** 与 **Production** 各开一套独立资源(尤其数据库),
> 避免压测/预览数据污染生产。

---

## 二、环境变量清单

在 Vercel 项目 **Settings → Environment Variables** 中按环境(Production / Preview / Development)配置。
本地开发复制 `apps/web/.env.example` 为 `apps/web/.env` 填值。

| 变量                         | 必填           | 示例 / 说明                                                          |
| ---------------------------- | -------------- | -------------------------------------------------------------------- |
| `DATABASE_URL`               | ✅             | `postgresql://user:pwd@host/db?sslmode=require`(Neon)                |
| `JWT_SECRET`                 | ✅             | 长随机串;生产务必从平台机密注入,勿入库                               |
| `JWT_TTL`                    | ⬜             | 访问令牌有效期,默认 `2h`                                             |
| `INITIAL_CHIPS`              | ⬜             | 新用户初始虚拟筹码,默认 `10000`                                      |
| `UPSTASH_REDIS_REST_URL`     | ✅             | `https://xxx.upstash.io`                                             |
| `UPSTASH_REDIS_REST_TOKEN`   | ✅             | Upstash Redis REST Token                                             |
| `MATCH_SIZE`                 | ⬜             | 快速匹配凑齐人数,默认 `4`                                            |
| `ABLY_API_KEY`               | ✅(实时里程碑) | Ably 应用的 API Key                                                  |
| `QSTASH_TOKEN`               | ✅(编排里程碑) | 发布 QStash 任务用                                                   |
| `QSTASH_CURRENT_SIGNING_KEY` | ✅(编排里程碑) | 回调验签(当前密钥)                                                   |
| `QSTASH_NEXT_SIGNING_KEY`    | ✅(编排里程碑) | 回调验签(轮换密钥)                                                   |
| `APP_URL`                    | ✅             | 应用对外根地址,如 `https://<your-app>.vercel.app`;用于拼 QStash 回调 |

> `ABLY_API_KEY` / `QSTASH_*` / `APP_URL` 在对应里程碑(实时、编排)接口落地后必填;
> M2/M3 阶段(认证/房间/匹配)仅需 `DATABASE_URL`、`JWT_SECRET`、`UPSTASH_REDIS_REST_*`。

## 三、Vercel 项目设置(Monorepo 关键)

1. **Import** 仓库到 Vercel,新建 Project。
2. **Root Directory** 设为 `apps/web`(关键)。Vercel 会据此自动套用 Next.js 预设,
   并在 Monorepo 中以 Turborepo 推断构建过滤。
3. **Framework Preset**:Next.js(自动识别)。
4. **Build/Install Command**:保持默认即可(Vercel 检测到 pnpm + Turborepo 会用
   `pnpm install` 安装、`next build` 构建)。`apps/web/vercel.json` 已声明
   `framework: nextjs`、函数 `maxDuration` 与示例 cron。
5. **Node 版本**:仓库 `engines.node >= 22`,在 Project Settings 选 Node 22。
6. **Package Manager**:pnpm(由 `packageManager` 字段 `pnpm@10.33.0` 锁定)。

> 已在根 `turbo.json` 用 `outputs: [".next/**", "!.next/cache/**"]`,符合 Vercel 对
> Next.js + Turborepo 的缓存建议。

## 四、数据库迁移(Neon + drizzle-kit)

迁移**不在 Vercel 构建期自动跑**(构建环境通常不应改生产库),手动按环境执行:

```bash
# 在本地或 CI,指向目标环境的 DATABASE_URL
export DATABASE_URL="postgresql://...sslmode=require"   # 目标库(Preview 或 Production)

# 生成迁移(改了 schema 后)
pnpm --filter @bullfighting/db drizzle-kit generate

# 应用迁移到目标库
pnpm --filter @bullfighting/db drizzle-kit migrate
```

要点:

- Neon 连接串务必带 `?sslmode=require`。
- Preview 与 Production 各自迁移;先在 Preview 验证再上 Production。
- 切勿把生产 `DATABASE_URL` 写进仓库或 CI 明文日志;用 CI Secret 注入。

> 具体脚本名以 `packages/db` 内 `drizzle.config` 与 package.json 脚本为准;
> 若已封装 `db:generate` / `db:migrate`,优先用封装命令。

## 五、QStash 回调与签名

- **回调地址**:`${APP_URL}/api/internal/qstash/advance`
  (例如 `https://<your-app>.vercel.app/api/internal/qstash/advance`)。
- 发布任务时用 `QSTASH_TOKEN` 鉴权;QStash 会带 `Upstash-Signature` 头回调。
- 回调路由**必须用 `QSTASH_CURRENT_SIGNING_KEY` / `QSTASH_NEXT_SIGNING_KEY` 验签**,
  拒绝伪造请求(详见 `docs/安全.md` §四)。回调需**幂等**防重放。
- Preview 与 Production 的回调地址不同,确保 `APP_URL` 随环境正确设置。

---

## 六、Ably(实时)

- 在 Ably 控制台创建 App,取 `ABLY_API_KEY` 配置到 Vercel。
- 客户端不直接持有 API Key;通过 `/api/realtime/token` 由服务端按当前用户
  **限定频道能力(capability)**颁发 token。
- 私有手牌走**单播私有频道**,绝不进广播(详见 `docs/安全.md` §三)。

---

## 七、安全响应头与 cron

- 安全响应头由 `apps/web/src/middleware.ts` 注入(不改 `next.config.mjs`),
  含 `X-Content-Type-Options`、`X-Frame-Options`、`Referrer-Policy`、CSP、HSTS 等。
- `apps/web/vercel.json` 含一个**每分钟示例 cron**,当前指向 `/api/health`
  作为最细粒度(Vercel cron 最细 1 分钟)的存活探针示例。
  待"对局清理"内部路由(如 `/api/internal/cleanup`,清理过期房间/超时回合)落地后,
  把 cron 的 `path` 改指向它即可:

  ```json
  {
    "crons": [{ "path": "/api/internal/cleanup", "schedule": "* * * * *" }]
  }
  ```

  注意:Vercel cron 仅在 **Production** 部署生效。

---

## 八、健康检查

- 端点:`GET /api/health` → `{ "status": "ok", ... }`(`runtime=nodejs`,`force-dynamic`)。
- 故意不触达数据库/Redis/Ably,作为轻量存活探针,不因下游抖动误报。
- 可用于外部 Uptime 监控与上面 cron 的示例目标。

## 九、Preview / Production 流程

1. **Preview**:对每个 PR / 非生产分支,Vercel 自动生成 Preview 部署。
   - 配置 Preview 环境变量(独立 Neon/Upstash/Ably 资源)。
   - 对 Preview 跑 E2E(`E2E_BASE_URL=<preview-url>`)与 k6 压测(见
     `e2e/README.md`、`infra/load/README.md`)。
2. **Production**:合并到 `main` 触发 Production 部署(或在 Dashboard Promote)。
   - 上线前过一遍 `docs/安全.md` §十 检查表。
   - 确认已对 Production 库执行迁移、`APP_URL` 指向生产域名、cron 仅生产生效。
