# 斗牛卡牌游戏 — 详细开发计划(定稿)

> 版本:v1.0　|　玩法:抢庄斗牛　|　筹码:虚拟娱乐筹码　|　技术栈:全栈 TypeScript　|　部署:Vercel

本文档为正式开发蓝图。已根据评审确认的 4 项关键决策定稿,落地为 **Vercel 原生 Serverless 全栈 TypeScript** 方案。

## 〇、玩法规则基线(抢庄斗牛)

所有逻辑严格依据此规则实现,实现于纯函数引擎 `@bullfighting/core`。

- **牌组**:标准 52 张(无大小王)。点数:A=1,2–9 按面值,10/J/Q/K 均记为 10。
- **每人 5 张**,从中找 **3 张点数之和为 10 的倍数**(`% 10 == 0`)即"成牛";剩余 2 张之和 `% 10` 决定**牛几**(为 0 即"牛牛",最大;无任何 3 张组合成牛即"没牛")。多种组合时取最优(牛值最大)。
- **特殊牌型**(可配置,默认开启):
  - **五小牛**:5 张点数均 ≤ 4 且总和 ≤ 10。
  - **四炸(炸弹)**:四张同点。
  - **五花牛**:5 张全为 J/Q/K。
- **牌型大小(默认,可配置)**:`五小牛 > 四炸 > 五花牛 > 牛牛 > 牛9 > … > 牛1 > 没牛`。同型比最大单张(先比点数 A<…<K,再比花色 ♠>♥>♣>♦);四炸比四条点数。
- **倍率表(默认,可配置)**:没牛~牛6=1,牛7=2,牛8=2,牛9=3,牛牛=3,五花牛=4,四炸=4,五小牛=5。
- **流程**:抢庄 → 下注 → 发牌 → 亮牌比牌 → 结算。庄家逐一对比每个闲家。
- **抢庄**:玩家以倍数竞价抢庄,最高者为庄(同倍可随机/先到);庄家倍数参与结算放大。
- **结算公式(零和)**:对每个闲家 p,`unit = 底分 × 闲家下注倍数 × 庄家抢庄倍数`,`金额 = unit × 赢家牛型倍率`;庄赢则庄家 +金额、该闲家 -金额,反之亦然。庄家净额 = -Σ各闲家净额,全局筹码守恒。
- **合规**:虚拟娱乐筹码,不可兑换现金,不涉及支付/KYC/博彩合规。

## 一、项目架构设计

**整体形态**:客户端–服务端,**服务端权威(Server-Authoritative)**——发牌、判牌、结算只在服务端进行,客户端只负责展示与输入,杜绝改牌作弊。

**核心架构决策**:斗牛规则做成**纯函数引擎包 `@bullfighting/core`**——无网络、无数据库、无副作用。好处:可被百万级组合穷举单测、与传输/存储解耦、天然防作弊、易复用(含未来 AI 机器人)。该决策使 Vercel Serverless 落地"零改动复用引擎"。

**Monorepo 结构**(pnpm workspaces + Turborepo):

```
bullfighting-poker/
├── packages/
│   ├── core/         # 斗牛规则引擎(纯 TS,零运行时依赖,可独立单测)
│   │   ├── points / rng / deck / evaluator / comparator / settlement / config
│   └── shared/       # 前后端共享:事件协议、DTO、类型(后续里程碑)
├── apps/
│   └── web/          # Next.js(前端 + Serverless API,后续里程碑)
├── infra/            # Vercel / CI 配置
└── docs/             # 文档(含本计划)
```

**部署拓扑(Vercel 原生)**:

```
┌──────────────────────────────────────────────┐
│        浏览器 (Next.js 前端 + Ably Client)       │
└───────┬───────────────────────────┬────────────┘
   HTTPS│ 行动:下注/抢庄/亮牌          │ WSS(由 Ably 托管)
        ▼                           ▲ 事件推送
┌──────────────────────────┐        │
│  Vercel Serverless 函数    │────────┘ (服务端经 Ably REST 发布)
│  Route Handlers + 状态机   │
│  @bullfighting/core 引擎    │◀──── QStash 延时回调(回合超时/推进)
└───┬───────────────┬────────┘
    ▼               ▼
┌─────────┐   ┌──────────────┐
│ Neon PG │   │ Upstash Redis │
│用户/战绩/│   │房间态/队列/锁  │
│ 账本     │   │               │
└─────────┘   └──────────────┘
```

**语言选型**:全栈 TypeScript。实时对局为 I/O 密集型,Node/TS 适配;前后端 + 引擎共用类型与协议,降低联调出错;生态成熟。

## 二、技术栈选择(Vercel 原生 · 全栈 TS)

| 层            | 选型                                              | 说明                                                 |
| ------------- | ------------------------------------------------- | ---------------------------------------------------- |
| 全栈框架      | Next.js 15(App Router)                            | 前端 + Serverless API 一体,Vercel 一键部署           |
| 实时推送      | Ably(推荐)/ Pusher / Supabase Realtime            | 托管 WebSocket;客户端订阅房间频道,服务端经 REST 发布 |
| 客户端行动    | Next.js Route Handlers(Serverless)                | 下注/抢庄/亮牌 → 服务端权威校验                      |
| 回合定时      | Upstash QStash(延时回调)+ Vercel Cron(粗粒度清理) | "N 秒后回调推进阶段/超时托管"                        |
| 关系库        | Vercel Postgres(Neon)                             | Serverless PG,强事务保证结算                         |
| KV / 房间态   | Upstash Redis                                     | 房间快照、匹配队列、结算分布式锁                     |
| ORM           | Drizzle                                           | Serverless 友好、轻量,配 Neon 驱动                   |
| 前端状态/动画 | Zustand + Framer Motion                           | 牌桌动画                                             |
| 规则引擎      | `@bullfighting/core`(纯 TS)                       | 服务端 Serverless 函数中运行                         |
| 测试          | Vitest + Playwright + k6                          | 单测/E2E/压测                                        |
| CI/CD         | GitHub Actions + Vercel                           | 门禁 + 自动部署/预览                                 |
| 监控          | Vercel Analytics + Sentry                         | 指标与错误追踪                                       |

> **Vercel 约束(已查证)**:不支持常驻 WebSocket;函数短时执行;Cron 最细 1 分钟。故实时改用托管服务,定时改用 QStash 延时回调 + Redis deadline 校验。

## 四、数据库设计

**PostgreSQL 主要表**:

| 表              | 关键字段                                                                                                                           | 说明                |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| `users`         | id, username(唯一), password_hash, nickname, avatar_url, chips(BIGINT), status, created_at                                         | 用户与筹码余额      |
| `user_stats`    | user_id(PK/FK), rounds_played, rounds_won, banker_rounds, total_won, total_lost, biggest_win                                       | 个人统计            |
| `rooms`         | id, room_code, name, owner_id(FK), base_score, max_players, mode, min_chips, status, created_at, closed_at                         | 房间与配置          |
| `room_seats`    | id, room_id(FK), user_id(FK), seat_no, status, chips_in, joined_at                                                                 | 座位/在座玩家       |
| `rounds`        | id, room_id(FK), round_no, banker_user_id(FK), shuffle_seed, shuffle_proof, phase, started_at, ended_at                            | 每一局              |
| `round_players` | id, round_id(FK), user_id(FK), seat_no, cards(JSONB), niu_type, niu_value, is_banker, bet_multiplier, rob_multiplier, result_chips | 局内每玩家结果      |
| `transactions`  | id, user_id(FK), round_id(FK,可空), type, amount(±), balance_after, created_at                                                     | 筹码账本(审计/对账) |

**关系**:`users 1—N room_seats N—1 rooms`;`rooms 1—N rounds`;`rounds 1—N round_players`;`users 1—N transactions`。

**关键约束**:结算时 `round_players` 写入 + `users.chips` 更新 + `transactions` 记账在**同一事务**内,涉及用户加锁,保证筹码守恒。

**Redis 结构**:

| Key                     | 类型       | 用途                         |
| ----------------------- | ---------- | ---------------------------- |
| `session:{token}`       | string     | 会话/在线态                  |
| `room:{id}:state`       | json       | 房间+局快照(高频读写,权威态) |
| `mm:queue:{baseScore}`  | zset       | 匹配队列                     |
| `lock:settle:{roundId}` | string(NX) | 结算分布式锁,防重复结算      |

| GET  | `/api/rooms/:id`               | 房间详情                  | 是   |

| POST | `/api/game/:roomId/bet`        | 闲家下注                  | 是   |
| POST | `/api/game/:roomId/reveal`     | 亮牌                      | 是   |
| POST | `/api/game/:roomId/sync`       | 断线重连拉快照            | 是   |
| POST | `/api/internal/qstash/advance` | QStash 回调推进阶段(内部) | 签名 |

**实时事件(Ably 频道 `room:{id}` 下发)**:

| 事件                                      | Payload                  | 说明                             |
| ----------------------------------------- | ------------------------ | -------------------------------- |
| `room:state`                              | 房间快照                 | 全量/增量同步                    |
| `game:phase`                              | `{phase, deadline}`      | 阶段切换 + 倒计时                |
| `game:deal`                               | `{cards}`                | **仅发给本人自己的牌**(私有频道) |
| `game:banker`                             | `{bankerId, multiplier}` | 庄家确定                         |
| `game:result`                             | 各家牌型                 | 亮牌结果                         |
| `game:settlement`                         | 各家盈亏 + 新余额        | 结算                             |
| `room:player_joined` / `room:player_left` | `{userId}`               | 玩家进出                         |

> 安全:`game:deal` 走玩家私有频道单播,比牌前服务端不下发他人手牌。

## 八、部署方案(Vercel)

- **一个 Vercel 项目**(Next.js),环境变量接 Ably / Upstash / Neon 密钥。
- **Preview Deployment**:每个 PR 自动生成独立预览环境,push 即验证;合并 `main` 即生产部署。
- **CI**:GitHub Actions 跑 lint/typecheck/测试门禁;Vercel 负责构建与部署。
- **Serverless 注意**:数据库走连接池/HTTP 驱动(Neon serverless driver),避免连接爆炸;无状态化,状态入 Redis;回合定时用 QStash 延时回调 + Redis deadline。
- **可观测性**:Vercel Analytics + Sentry + 结构化日志。
- **演进**:若未来需超高并发/极致低延迟,可迁移为常驻 Node + Socket.IO 专用实时后端;因引擎为纯函数,迁移成本低。
