# 斗牛扑克 · 上手部署指引(保姆级)

> 目标:从零把项目部署到 **Vercel** 并跑起来。
> 这是**一步步照着做**的操作手册;原理与权衡见参考文档 [`部署.md`](./部署.md) 与 [`安全.md`](./安全.md)。
>
> 形态:pnpm + Turborepo Monorepo,单个 Next.js 15 应用(`apps/web`)。
> 依赖 4 个免费托管服务:**Neon**(Postgres)、**Upstash Redis**、**Upstash QStash**、**Ably**(实时)。

---

## 0. 总览:你要拿到的 12 个环境变量

| 变量 | 来自 | 必填 | 说明 |
| --- | --- | :---: | --- |
| `DATABASE_URL` | Neon | ✅ | Postgres 连接串(带 `?sslmode=require`) |
| `JWT_SECRET` | 本地生成 | ✅ | 登录态签名密钥(长随机串) |
| `UPSTASH_REDIS_REST_URL` | Upstash Redis | ✅ | REST 接口地址 |
| `UPSTASH_REDIS_REST_TOKEN` | Upstash Redis | ✅ | REST 接口令牌 |
| `QSTASH_TOKEN` | Upstash QStash | ✅ | **必须是可发布的主令牌**(见 §3 的坑) |
| `QSTASH_CURRENT_SIGNING_KEY` | Upstash QStash | ✅ | 回调验签(当前) |
| `QSTASH_NEXT_SIGNING_KEY` | Upstash QStash | ✅ | 回调验签(轮换) |
| `ABLY_API_KEY` | Ably | ✅ | 实时推送密钥 |
| `APP_URL` | Vercel(部署后) | ✅ | 应用对外根地址,**部署后回填**(见 §7) |
| `INITIAL_CHIPS` | — | ⬜ | 新用户初始筹码,默认 `10000` |
| `MATCH_SIZE` | — | ⬜ | 快速匹配人数,默认 `4` |
| `JWT_TTL` | — | ⬜ | 登录态有效期,默认 `2h` |

> 模板见 `apps/web/.env.example`。本地开发时复制为 `apps/web/.env` 再填值。
> ⚠️ **真实值绝不写进仓库**,只放进 Vercel 环境变量。

## 1. Neon —— `DATABASE_URL` + 建表

1. 打开 <https://console.neon.tech> → 登录 → **Create Project**(Region 建议 `ap-southeast-1 / Singapore`)。
2. 项目首页 **Connection string** 区块,选 **Pooled connection**,复制那串
   `postgresql://...-pooler...neon.tech/neondb?sslmode=require...` → 就是 `DATABASE_URL`。
3. **建表(必做)**。两种方式任选其一:

   **方式 A:用迁移脚本(推荐)** —— 在本地仓库执行:
   ```bash
   export DATABASE_URL="postgresql://...sslmode=require"   # 目标库
   pnpm --filter @bullfighting/db db:migrate
   ```
   **方式 B:手动跑 SQL** —— 在 Neon 控制台 **SQL Editor** 里,依次粘贴执行:
   - `packages/db/migrations/0000_grey_valkyrie.sql`
   - `packages/db/migrations/0001_tiny_whirlwind.sql`

> Preview 与 Production 若用不同库,需各自建表一次。

---

## 2. Upstash Redis —— `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN`

1. 打开 <https://console.upstash.com> → 登录 → 顶部 **Redis** → **Create Database**。
2. Name 随意;Type 选 **Regional**;Region 尽量和 Neon/Vercel 同区(如 Singapore)。
3. 进数据库详情页 → 下拉到 **REST API** 区块 → 点 **`.env`** 小标签,会看到:
   ```
   UPSTASH_REDIS_REST_URL="https://xxxx.upstash.io"
   UPSTASH_REDIS_REST_TOKEN="AXXXAAIj..."
   ```
   两行的值对号入座填入同名变量。

---

## 3. Upstash QStash —— `QSTASH_TOKEN` + 两个签名密钥

> ⚠️ **最容易踩的坑**:QStash 有「只读令牌」和「主令牌」。游戏靠 QStash **发布(publish)**
> 定时回调来推进对局阶段,**只读令牌无法发布**,会导致超时不自动推进。**必须用主令牌。**

1. 同一个 Upstash 控制台,顶部切到 **QStash**。
2. 复制页面上的**主 `QSTASH_TOKEN`**(默认完整权限那个,**不要**复制任何标着 read-only 的)。
3. 在签名密钥区复制:
   ```
   QSTASH_CURRENT_SIGNING_KEY=sig_xxxx
   QSTASH_NEXT_SIGNING_KEY=sig_yyyy
   ```
4. **自检令牌是否可发布**(可选但强烈建议)——QStash token 是一段 base64 的 JSON,解出来
   `Type` 不能是 `read_only`:
   ```bash
   node -e "console.log(JSON.parse(Buffer.from(process.env.QSTASH_TOKEN,'base64').toString()).Type)"
   ```
   输出 `read_only` ❌ → 换主令牌;输出其它(如 `token`)✅。

---

## 4. Ably —— `ABLY_API_KEY`

1. 打开 <https://ably.com> → 注册/登录 → 控制台 **Create new app**。
2. 进 app → **API Keys** 标签 → 复制默认 Key 的完整值(形如 `appId.keyId:secret`)→ 就是 `ABLY_API_KEY`。

> 客户端**不直接持有**这个 Key;服务端 `/api/realtime/token` 按用户颁发受限 token,
> 私有手牌走单播私有频道(详见 `安全.md` §三)。

---

## 5. 生成 `JWT_SECRET`

本地任一命令生成一个强随机串:
```bash
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"
# 或:openssl rand -base64 48
```
把输出作为 `JWT_SECRET`。生产与预览建议各用一个不同的值。

---

## 6. Vercel —— 导入项目 + 配置环境变量

> 本项目是 Monorepo,**Root Directory 必须设成 `apps/web`**,否则构建必失败。

1. <https://vercel.com> → **Add New… → Project** → 选仓库 `chenbowen-cbw/bullfighting-poker`。
2. **Root Directory** → 点 **Edit** → 选 **`apps/web`**。Framework 会自动识别为 **Next.js**。
3. **Node 版本**:Project Settings 选 **Node 22**(仓库 `engines.node >= 22`)。
   Build / Install Command 保持默认(Vercel 会用 pnpm + Turborepo)。
4. 展开 **Environment Variables** → 直接把整段 `KEY=value` **批量粘贴**(Vercel 支持
   `.env` 格式整段导入,自动拆成多条)。把 §0 表里的变量都填上;`APP_URL` 先随便填个占位
   (如 `https://placeholder.vercel.app`),§7 再回填。
   - 环境建议三个都勾:**Production / Preview / Development**。
   - `# 可选` 这种注释行不要当变量贴。
5. 点 **Deploy**,等首次构建完成。

---

## 7. 回填 `APP_URL` 再部署一次(关键收尾)

首次部署成功后,Vercel 会分配真实域名。**QStash 定时回调要打到 `${APP_URL}/api/internal/qstash/advance`**,
所以 `APP_URL` 必须是真实地址,否则超时推进会失败。

1. 进项目 **Project → Domains / Deployments**,记下真实域名(如 `https://<your-app>.vercel.app`)。
2. **Settings → Environment Variables** → 把 `APP_URL` 改成这个真实地址 → 保存。
3. **Deployments** 页对最新部署点 **⋯ → Redeploy** 让新值生效。

## 8. 冒烟验证

- 浏览器开 `https://<your-app>.vercel.app/api/health` → 返回 `{"status":"ok",...}` 说明应用活着。
- 打开首页 → 注册两个账号(可用两个浏览器/无痕窗口)→ 建房 / 加入 → 房主开局。
- 走一整局:**抢庄 → 下注 → 亮牌 → 结算**,确认筹码变化、实时同步、超时能自动推进。
- 刷新页面后对局态能恢复(/sync),说明 Redis 状态正常。

## 9. 常见坑速查

| 现象 | 多半原因 |
| --- | --- |
| 构建失败、找不到 Next.js | Root Directory 没设成 `apps/web` |
| 登录就报错 / 立刻掉登录 | `JWT_SECRET` 没配,或 Preview/Prod 用了不同值后切换了环境 |
| 能操作但**超时不自动推进**阶段 | `QSTASH_TOKEN` 是只读令牌(见 §3),或 `APP_URL` 不对/没回填重部署 |
| 实时不同步、只能靠刷新 | `ABLY_API_KEY` 没配或无效;页面会降级为轮询 |
| 一操作就报数据库错误 | `DATABASE_URL` 没配 / 没建表(回到 §1 跑迁移) |
| 房间能建但匹配/状态异常 | `UPSTASH_REDIS_REST_URL/TOKEN` 没配 |

---

## 10. 安全红线(务必遵守)

- **任何密钥、连接串都不入仓库**(`.env` 已被忽略;仓库里只保留 `.env.example` 模板)。
- 不在日志/PR/截图里贴明文密钥。
- 一旦怀疑泄露,立即到对应服务**轮换(rotate)**密钥并更新 Vercel 变量后重部署。
- Vercel cron 仅在 **Production** 生效;Preview 与 Production 尽量用各自独立的三方资源。
